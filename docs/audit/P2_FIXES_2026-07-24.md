## P2-1: VanillaMachineBatchDelegate 输出槽退款守恒破洞

### 问题描述

**文件**：`src/main/java/com/huanghuang/rsintegration/mods/vanilla/VanillaMachineBatchDelegate.java`  
**位置**：L798-802（修复前）

**原代码**：
```java
ItemStack slot2 = furnaceBE.getItem(2);
if (!slot2.isEmpty()) {
    furnaceBE.setItem(2, ItemStack.EMPTY);
    if (refundToRS) refundToRSNetwork(slot2);  // ❌ 问题：输出槽被退款
}
```

**问题分析**：
- 输出槽（slot 2）存放的是**已转化的产物**（例如铁矿石 → 铁锭）
- 产物不是 ledger 管理的输入材料，不应该通过 `clearMachineState` 退款
- 产物应该只通过 `collectResult()` 收集
- 如果在 abort 路径退款，可能与 `collectResult` 竞态，导致产物被双重收集（双刷）

**影响场景**：
1. 熔炉烧炼完成，产物在 slot 2
2. 玩家离线，batch 被 silent abort
3. `clearMachineState(be, null)` 调用，`refundToRS = true`
4. 输出槽产物被退还到网络
5. 如果 `collectResult` 也执行，产物被双重收集

### 修复方案

**修复后代码**：
```java
// Output slot (slot 2): the transformed result is not a ledger-managed input.
// On abort, clear it to prevent residue, but do NOT refund — collectResult()
// is the only path that should collect the output. Refunding here would risk
// double-collection if collectResult() already ran or races with cleanup.
ItemStack slot2 = furnaceBE.getItem(2);
if (!slot2.isEmpty()) {
    furnaceBE.setItem(2, ItemStack.EMPTY);
    // Do NOT refund output slot to network (no `if (refundToRS) refund...`)
}
```

**修复要点**：
1. 清空输出槽防止残留
2. **不退款**，避免双重收集
3. 添加详细注释说明原因

**验证方法**：
1. 设置熔炉自动烧炼配方
2. 烧炼进行中途，玩家离线（触发 silent abort）
3. 检查产物是否被重复计入网络（预期：否）

## P2-2: CookingPotBatchDelegate 容器退款注释不清

### 问题描述

**文件**：`src/main/java/com/huanghuang/rsintegration/mods/farmersdelight/CookingPotBatchDelegate.java`  
**位置**：L487-488（修复前）

**原代码**：
```java
ItemStack container = handler.extractItem(CONTAINER_SLOT, 64, false);
if (!container.isEmpty() && !usingSharedLedger) refundToRSNetwork(container);
```

**问题分析**：
- 容器（碗/杯等）通过 `getSupplementalSpecs()` 声明，属于 shared ledger 管理的材料
- 在 `tryStartWithMaterials` 中从 materials 列表分离（L204-216），但仍在 ledger 记录中
- 当前守卫逻辑 `!usingSharedLedger` 是**正确的**，但缺少注释说明
- 容易在后续维护中误解容器来源，导致错误修改

**对比参考**：
EnchantalCoolerBatchDelegate 的容器是 **out-of-band** 材料（在方法内部直接从 RS 提取，不在 materials 中），所以无条件退款：
```java
// Container is out-of-band (not in shared ledger) — refund unconditionally
ItemStack container = handler.extractItem(CONTAINER_SLOT, 64, false);
if (!container.isEmpty()) refundToRSNetwork(container);
```

### 修复方案

**修复后代码**：
```java
// Container was reserved via the shared ledger (separated from materials in
// tryStartWithMaterials L204-216). When usingSharedLedger=true, the ledger's
// refundCommitted() will restore it; delegate must not double-refund.
ItemStack container = handler.extractItem(CONTAINER_SLOT, 64, false);
if (!container.isEmpty() && !usingSharedLedger) refundToRSNetwork(container);
```

**修复要点**：
1. 添加详细注释说明容器来源（从 materials 列表分离）
2. 说明退款责任（shared ledger 负责，delegate 不应重复退款）
3. 保持现有守卫逻辑不变

**澄清**：
- 当 `usingSharedLedger=true`：ledger 的 `refundCommitted()` 会退款
- 当 `usingSharedLedger=false`：delegate 手动退款

## 统一退款守卫模式建议

基于修复经验，建议所有 delegate 采用以下统一模式：

```java
private void clearMachineState(BlockEntity be, ServerPlayer player) {
    // 输入材料：由 ledger 管理，只在私有 ledger 且 player==null 时退款
    if (!usingSharedLedger && player == null) {
        refundInputMaterials();
    }
    
    // Out-of-band 材料（燃料/容器等，不在 ledger 中）：总是退款
    refundOutOfBandMaterials();
    
    // 输出产物：从不在 clearMachineState 中退款（只通过 collectResult）
    clearOutputSlots();  // 清空但不退款
}
```

**关键原则**：
1. **Ledger 管理的材料**：由 ledger 负责退款，delegate 根据 `usingSharedLedger` 决定是否手动退款
2. **Out-of-band 材料**：不在 ledger 中，总是由 delegate 退款
3. **输出产物**：已转化的产物，不是输入材料，不退款（只通过 collectResult 收集）
