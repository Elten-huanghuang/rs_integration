# RS 终端神化刷怪笼升级功能计划书

## 实现状态（2026-07-24）

已完成第一阶段可运行闭环：

- 手持有效 RS 终端对神化刷怪笼蹲下右键打开专用界面；
- 从服务端当前加载的 `spawner_modifier` 配方发现全部正向升级；
- 读取真实当前值、上下限、应用次数、Ingredient 和 RS 库存；
- 升级项目/材料需求双视图、选择保留、已生效禁用和刷新；
- 每个升级项逐次使用独立 `ExtractionLedger`，单项缺料或失败不阻断后续项；
- 通过 Apotheosis `SpawnerModifier.apply` 修改状态，不直接写 NBT；
- 中英文界面文本、网络包边界校验、完整测试与发布构建通过。

已接入多根材料需求的 `CraftingResolver`、现有 `CraftingPlanScreen` DAG 预览和 `AsyncCraftChain` 执行链。升级界面的执行按钮只请求服务端规划，不会直接启动合成；玩家在配方树中确认后，服务端才执行短期预览会话。会话绑定玩家、维度、刷怪笼坐标和勾选数量，超时或位置变化时拒绝执行。每个升级材料根独立重算：可递归的根按稳定顺序逐个执行，每条链完成后使用最新库存规划下一根；完全无配方的根保留为缺料，不阻断其他根。全部可执行递归链结束后再次验证刷怪笼、网络和选择次数，再应用已有材料的升级。

## 1. 目标

玩家手持已连接 RS 网络的终端右键神化刷怪笼时，打开专用升级界面。玩家可以选择需要的正向升级，查看当前值、目标值、材料需求和说明；库存不足时可以进入缺料/配方树视图递归合成；执行时各升级项相互独立，某一项失败不能阻断其他可执行项。

第一版即覆盖 Apotheosis 1.20.1 的全部正向刷怪笼升级，不只覆盖时间和红石控制。

## 2. 非目标

- 不提供反向升级自动化。需要副手石英的 inverted 配方不出现在选择列表中。
- 不直接写刷怪笼 NBT 或绕过 Apotheosis 的属性修改逻辑。
- 不允许远程升级未加载区块、已被替换的方块或玩家无权交互的方块。
- 不把所有勾选项视为必须同时成功的一笔交易。
- 不模拟客户端点击、临时修改玩家主副手或生成假玩家。

## 3. 支持范围

以服务端当前 `RecipeManager` 中加载的 `apotheosis:spawner_modifier` 配方为准，兼容数据包对材料与上下限的修改。Apotheosis 7.4.3 默认正向升级如下：

| 分类 | 属性 | 默认材料 | 默认效果 | 默认极限 |
|---|---|---|---|---|
| 速度 | 最小生成延迟 | 糖 | 每次 -10 tick | 20 tick |
| 速度 | 最大生成延迟 | 时钟 | 每次 -10 tick | 20 tick |
| 数量 | 每轮生成数量 | 发酵蜘蛛眼 | 每次 +1 | 16 |
| 数量 | 附近实体上限 | 恶魂之泪 | 每次 +2 | 32 |
| 范围 | 玩家激活范围 | 海晶砂粒 | 每次 +2 | 48 |
| 范围 | 生物生成范围 | 烈焰棒 | 每次 +1 | 32 |
| 行为 | 忽略附近玩家要求 | 下界之星 | 设为启用 | 一次性 |
| 行为 | 忽略生成条件 | 龙蛋 | 设为启用 | 一次性 |
| 行为 | 忽略光照 | 灵魂灯笼 | 设为启用 | 一次性 |
| 行为 | 红石控制 | 红石比较器 | 设为启用 | 一次性 |
| 生物 | 无 AI | 紫颂果 | 设为启用 | 一次性 |
| 生物 | 静音 | 任意羊毛 | 设为启用 | 一次性 |

## 4. 交互入口

触发条件：

1. 玩家主手或副手持有 RSI 已支持的 RS 网络终端。
2. 玩家按住潜行键（蹲下）再右键；普通右键不触发本功能。
3. 右键目标方块实体是 `ApothSpawnerTile`。
4. 对应终端能解析出玩家有权限访问的在线 RS 网络。
5. 玩家对目标位置具有正常交互权限，且目标区块已加载。

满足条件时服务端拦截本次方块右键并打开升级界面。否则保持终端和方块原本行为。不能扩大现有 `NetworkItemUseMixin` 的全局拦截范围，避免破坏普通远程终端 GUI。

刷怪笼不创建绑定记录、不经过绑定 HUD，也不需要绑定/解绑流程。不能扩大现有 `NetworkItemUseMixin` 的全局拦截范围，避免破坏普通远程终端 GUI。

## 5. 界面设计

### 5.1 升级选择视图

这是默认视图。顶部固定显示刷怪笼实体图标、名称、目标坐标和网络连接状态。主体为紧凑列表，按“速度、数量、范围、行为、生物”分组，不使用嵌套卡片。

每行包含：

- 复选框；
- 升级材料图标；
- 属性名称；
- 当前值到目标值；
- 需要次数与材料总量；
- 一行简短效果说明；
- 状态标记。

状态必须互斥且明确：

- `可直接升级`：RS 网络现有材料足够；
- `可合成`：库存不足，但递归配方计划可完成；
- `缺少材料`：递归计划仍有原料缺口；
- `已生效`：布尔属性已经启用；
- `已达上限`：数值属性已达到配方边界；
- `不适用`：本次属性变化不会改变该刷怪笼；
- `配方不支持`：配方结构超出当前安全执行能力。

`已生效`、`已达上限`和`不适用`行不可勾选，也不能消耗材料。默认不勾选任何升级项；玩家可逐项选择，或主动点击“全选可用”；玩家选择在本次界面会话中保留。

数值升级使用行内滑条选择本次应用次数，范围为 0 到当前状态可达到配方上限所需的次数；界面同步预览执行后的属性值和材料数量。无 AI、红石控制等一次性布尔升级使用自绘开关。所有页面命令按钮使用 RSI 自绘扁平按钮，不使用原版按钮皮肤。

底部固定操作区：

- `查看材料`：切换到材料与配方树视图；
- `执行已选`：执行所有当前可执行的已选项；
- `关闭`。

执行按钮旁显示汇总，例如“9 项可执行，2 项缺料，3 项已完成”。不可用项目不使整个按钮失效。

### 5.2 材料与配方树视图

该视图只汇总当前勾选且尚未完成的升级，不重复展示已完成项目。顶部提供分段控件在“材料”和“配方树”之间切换，并提供返回升级选择视图的按钮。

材料模式显示：

- 每种最终升级材料的需求、网络库存与缺口；
- 递归合成后的基础原料总需求；
- 哪些升级项使用该材料；
- 单项取消选择入口。

配方树模式复用现有 `CraftingPlanScreen` 的树模型、布局、替代配方选择和预览组件，但使用独立的刷怪笼会话容器，不能把多个升级材料伪装成一个普通输出配方。每个缺失材料是一个计划根；相同子配方仍按现有等价节点聚合规则合并展示。不新增 JEI 自定义加号入口，配方树只从本界面的“查看材料”进入。

两个主视图来回切换时必须保留：

- 已勾选项目；
- 每项计算状态；
- 配方替代项选择；
- 树的折叠状态；
- 当前滚动位置。

网络库存变化后只刷新状态和数量，不重置玩家选择。刷新应有请求编号，过期响应不得覆盖新状态。

## 6. 计划与执行模型

### 6.1 数据模型

服务端创建短生命周期的 `SpawnerUpgradeSession`：

- 玩家 UUID；
- 维度与刷怪笼坐标；
- 初始方块实体身份/状态摘要；
- RS 网络身份；
- 已加载正向 modifier 列表；
- 每项当前值、目标值、所需应用次数；
- 每项材料计划与递归合成计划；
- 客户端选择集和请求序号；
- 创建时间与最后活动时间。

会话只保存计划，不预扣库存。执行前必须基于最新世界、配方和网络状态重新验证。

### 6.2 modifier 发现

从服务端 `RecipeManager` 获取 Apotheosis modifier 配方：

1. 排除带副手石英条件的 inverted 配方。
2. 保留主手 Ingredient、属性变化、是否消耗副手等原始定义。
3. 根据 `StatModifier` 的方向和当前值计算达到边界需要应用多少次。
4. 多属性 modifier 作为一个不可拆分升级项显示和执行。
5. 无法安全判定方向、边界或幂等性的自定义配方标记为“不支持”，不能猜测执行。

默认配方均属于可支持范围。不能只按 recipeId 后缀判断正反向，最终应以副手要求和属性变化语义共同校验。

### 6.3 材料代表选择

Ingredient 可能是标签，例如静音使用 `minecraft:wool`。选择实际消耗物品时依次优先：

1. RS 网络中库存足够的成员；
2. 递归计划能实际产出的成员；
3. 库存最多的成员；
4. 配方 Ingredient 的第一个合法成员，仅用于显示，不能据此宣称可执行。

这与普通制作计划的 tag 代表选择规则保持一致，避免界面显示一种羊毛而实际计划制作另一种。

### 6.4 部分成功

每个升级项是独立事务，执行顺序固定为：速度、数量、范围、行为、生物；同组按 recipeId 排序，确保结果可复现。

对每一项：

1. 重新确认刷怪笼仍存在且玩家仍有权限。
2. 重新计算该 modifier 是否还能改变状态及剩余次数。
3. 准备该项所需材料；需要时启动并等待该项递归合成。
4. 每次应用前，从 RS 网络通过独立 `ExtractionLedger` 预留一次材料。
5. 调用 Apotheosis 自己的 `SpawnerModifier.apply(tile)`。
6. 仅当 `apply` 返回 true 时提交该次材料消耗。
7. 返回 false、目标失效或发生异常时回滚该次预留，停止该项并继续下一项。
8. 成功后标记方块实体已更改并发送方块更新。

一个升级项缺料、无配方或合成失败，只改变该项状态。其他勾选项继续运行。最终结果应显示“成功 X 项、跳过 Y 项、失败 Z 项”，并允许保留失败项后返回配方树。

### 6.5 异步递归合成

不能在一次网络包处理或服务器 tick 内同步完成整批递归制作。升级任务应作为服务端异步链运行，并复用现有 `AsyncCraftChain` 的配方解析、机器等待、超时、退款和断线恢复能力。

建议采用“按升级项顺序准备并提交”的模型，而不是先制作全部材料：

- 已完成项目更早产生可见结果；
- 中途失败不会积压后续升级材料；
- 每项账本边界清晰；
- 玩家关闭 GUI 后服务器仍可按策略继续或安全停止。

执行期间客户端显示每项 `等待材料 / 合成中 / 应用中 / 完成 / 失败`，不允许重复提交同一会话。

## 7. 权限、安全与一致性

- 所有状态读取、规划确认、提取和属性修改均在服务端执行。
- 客户端只发送会话 ID、请求序号、选择项 ID 和配方选择，不发送可信数量或属性目标。
- 打开和执行时都调用 Forge 方块交互/保护兼容检查；不能仅依赖距离检查。
- 玩家必须仍在允许交互距离内，除非未来明确设计远程升级权限。
- 会话绑定玩家、维度、位置和网络，其他玩家不能复用。
- 方块被拆除、替换、区块卸载或刷怪笼 NBT 身份变化时停止后续操作并退款未提交预留。
- 配方重载后使旧会话失效，要求重新打开界面。
- 包请求需要频率限制和最大选择数限制。
- `apply` 之前再次检查当前状态，避免两个玩家同时升级造成重复扣料。
- 任何一次材料只能由一个 ledger 所有；禁止同时由总账本和升级项账本退款。

## 8. 兼容策略

Apotheosis 作为可选依赖。所有直接引用其类的实现必须放在 `mods.apotheosis` 范围，并由 `ModList` 与 `ENABLE_APOTHEOSIS` 双重守卫。没有安装或模块关闭时不能类加载失败。

优先使用公开接口：

- `SpawnerModifier.getMainhandInput()`；
- `SpawnerModifier.getOffhandInput()`；
- `SpawnerModifier.getStatModifiers()`；
- `SpawnerModifier.apply(ApothSpawnerTile)`；
- Apotheosis 的 modifier recipe type。

不直接访问混淆字段，不写内部刷怪笼 NBT。若未来 Apotheosis API 变化，应让兼容层拒绝开启功能并记录一次明确错误，而不是继续猜测执行。

资源包可提供已知属性的中文名称和说明。未知数据包属性使用其 ID 展示，并默认不允许自动执行，直到兼容层能可靠计算变化边界。

## 9. 客户端表现

- 使用现有 RSI 深灰、绿色强调色体系，同时用绿色、琥珀色、红色区分可执行、等待和失败。
- 行高固定，图标和状态变化不能导致布局跳动。
- 说明通过行内短句和物品悬浮提示呈现，不使用遮挡主体的大型浮层。
- 小分辨率下主体列表滚动，顶部标题与底部操作区固定。
- 执行时只更新发生变化的行，不反复关闭重开界面。
- 音效只在开始执行、单项完成和整批结束时播放；连续数值升级不能每消耗一个物品就播放一次。

## 10. 网络包建议

- `OpenSpawnerUpgradePacket`：服务端响应右键并创建会话。
- `SpawnerUpgradeStatePacket`：服务端下发当前属性、配方、材料与状态。
- `SpawnerUpgradePreviewPacket`：客户端提交选择/替代配方，服务端重新规划。
- `ExecuteSpawnerUpgradesPacket`：客户端提交最终选择。
- `SpawnerUpgradeProgressPacket`：服务端推送单项进度和最终结果。
- `CancelSpawnerUpgradePacket`：关闭界面时取消尚未开始的任务；已提交的单项按既定退款规则收尾。

所有包都必须验证方向、玩家、会话 ID、请求序号和负载上限。

## 11. 实施阶段

### 阶段 A：兼容与领域模型

- 建立可选加载的 Apotheosis 兼容层。
- 读取并分类全部正向 modifier 配方。
- 实现当前值、边界、所需次数与材料需求计算。
- 为默认 13 项升级建立本地化名称、说明和排序。

### 阶段 B：只读界面

- 实现终端右键入口和服务端会话。
- 完成升级选择视图、禁用状态和实时网络库存预览。
- 暂不执行，只验证不同刷怪笼状态下的计算结果。

### 阶段 C：材料计划与双视图

- 接入现有递归计划器。
- 实现材料汇总、多根配方树和两个视图间的状态保留。
- 支持 Ingredient 标签成员与替代配方选择。

### 阶段 D：执行与恢复

- 实现逐项异步执行和独立 ledger。
- 接入进度更新、部分成功、断线、区块卸载和方块替换处理。
- 确认同一刷怪笼并发升级不会重复扣料。

### 阶段 E：兼容与体验收尾

- 测试数据包修改过的材料、上下限和多属性 modifier。
- 完成本地化、窄屏布局、键鼠交互、提示与日志节流。
- 增加配置开关、操作审计日志和发布文档。

## 12. 测试清单

### 计算测试

- 默认刷怪笼到每项上限所需次数正确。
- 部分升级后的剩余次数正确。
- 布尔属性已启用时需求为零且不可选择。
- 数值已经越过或等于边界时不继续消耗。
- 数据包改变材料、步长或上限后按新配方计算。
- 羊毛标签按实际库存或计划产物选择成员。

### 事务测试

- 材料足够时全部选中可依次完成。
- 某一项缺料时其余项目继续完成。
- 递归合成某一项失败时已完成项不退款，未开始项不扣料。
- `apply` 返回 false 时本次材料完整回滚。
- 玩家断线、死亡、换维度和服务器关闭时没有复制或吞物。
- 刷怪笼被拆除、替换或区块卸载时未提交材料回滚。
- 两名玩家同时升级同一个刷怪笼不会重复消费或突破上限。

### 交互测试

- 右键普通方块仍打开终端原 GUI。
- 潜行右键绑定/解绑逻辑不受影响。
- 已完成项目不可勾选、不可执行。
- 两个视图来回切换不丢选择和树状态。
- 旧预览响应不能覆盖新选择。
- 低分辨率和 GUI 缩放下文字、图标、按钮不重叠。

### 兼容测试

- 未安装 Apotheosis 时正常启动。
- 关闭 `ENABLE_APOTHEOSIS` 时入口完全禁用。
- Apotheosis 刷怪笼模块关闭时不注册入口。
- 权限模组拒绝交互时不提取或合成材料。
- 配方重载后旧会话失效且无资源损失。

## 13. 验收标准

1. 默认 13 项正向升级均可正确显示、选择并达到各自配方上限。
2. 已生效或已达上限的升级不会再次消耗材料。
3. 任一项目缺料或失败不会阻断其他可执行项目。
4. 缺料可进入递归配方树，返回后选择与树状态不丢失。
5. 执行只使用服务端当前加载的 modifier 配方，并通过 Apotheosis 的 `apply` 修改刷怪笼。
6. 所有失败、断线和世界状态变化路径均无复制、吞物或双重退款。
7. 只有蹲下右键刷怪笼才进入升级界面；不影响终端普通右键、现有绑定/解绑和远程 GUI 功能。

## 14. 推荐决策

采用“全正向升级 + 数据驱动配方发现 + 已知属性白名单执行”的折中方案：界面从当前服务端配方读取真实材料和上限，但只有能可靠计算边界和验证幂等性的属性允许自动执行。这样默认 13 项可以一次到位，也不会因为整合包加入了结构未知的 modifier 就冒险扣料或写坏刷怪笼。

执行层必须坚持逐项独立事务。用户体验上是一键完成所有可完成项目，工程上仍然是多个可恢复、可跳过的小操作，这是“不因一个勾选项失败而打断整体流程”的核心保证。
