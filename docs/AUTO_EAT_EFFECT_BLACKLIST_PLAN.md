# 自动进食效果黑名单功能计划

## 1. 背景与目标

当前自动进食功能支持按具体物品设置黑名单，但无法表达“不要自动食用任何会施加某种状态效果的食物”。当整合包包含酒类、妖怪食物或带概率效果的料理时，玩家需要逐个屏蔽物品，维护成本较高。

本功能在现有黑名单界面中增加“状态效果”分类，让玩家可以直接屏蔽例如“酩酊”、反胃、中毒等效果。命中效果黑名单的食物不会被自动进食，但仍可由玩家手动食用。

首版目标：

- 保留现有物品黑名单行为；
- 支持按 `MobEffect` 注册名设置效果黑名单；
- 在 Diversity、Stack、Diet 三种自动进食模式中统一生效；
- 正确读取依赖当前玩家状态生成的 `FoodProperties`；
- 在原有黑名单界面中完成查看、搜索、添加和移除；
- 对无法通过标准食物属性预测的动态效果明确兼容边界。

## 2. 用户体验

### 2.1 入口

继续使用现有自动进食黑名单入口，不增加新的独立配置页面。进入后通过紧凑的分页或分段控件切换：

```text
自动进食黑名单

[ 食物 ] [ 状态效果 ]

[搜索名称或注册名                       ]

图标  名称                         状态
      酩酊                         已屏蔽
      反胃                         允许
```

实际界面使用 Minecraft 的物品图标和状态效果图标，示意文本中的符号不作为最终视觉资产。

### 2.2 操作规则

- “食物”页保留当前物品黑名单功能；
- “状态效果”页列出当前注册表中的全部 `MobEffect`；
- 点击列表项切换允许/屏蔽状态；
- 已屏蔽项优先显示，其余项目按本地化名称排序；
- 搜索同时匹配本地化名称和完整注册名；
- 悬停显示效果名称、注册名和当前状态；
- 再次点击已屏蔽效果即可移除；
- 修改结果通过现有或扩展后的网络消息提交给服务端；
- 服务端持久化并作为最终过滤权威。

### 2.3 运行时表现

当食物的标准效果列表包含任意黑名单效果时，自动进食跳过该食物。

例如效果黑名单包含 `youkaishomecoming:drunk` 时，所有在 `FoodProperties` 中声明该效果的酒类都不再进入自动进食候选集合。

规则只影响自动进食：

- 不修改食物本身；
- 不从食物中删除效果；
- 不阻止手动食用；
- 不改变其他模组的使用条件；
- 不模拟一次食用来预测结果。

如果全部候选食物都被过滤，应沿用现有无候选处理，并在适合的反馈位置区分“库存没有食物”和“食物被黑名单过滤”。首版可以先记录过滤原因，待现有反馈通道允许后再展示详细原因。

## 3. 视觉风格

界面延续现有自动进食界面与 Minecraft 容器 UI 的视觉语言，不引入网页式大卡片或独立设计体系。

### 3.1 布局

- 标题、背景、边距和关闭方式沿用现有黑名单界面；
- “食物/状态效果”使用稳定宽度的分段控件；
- 搜索框放在分类控件下方；
- 内容区使用紧凑网格或列表，保证大量模组效果下仍便于扫描；
- 控件尺寸不因名称长度、悬停或选中状态发生跳动；
- 在不同 GUI Scale 下保证文字、图标和点击区域不重叠。

### 3.2 状态表现

| 状态 | 建议表现 |
|---|---|
| 允许 | 正常图标和中性文字 |
| 悬停 | 浅色边框或原生高亮背景 |
| 已屏蔽 | 图标轻微变暗，并显示明确的红色禁用标记 |
| 搜索无结果 | 内容区显示简短空状态 |
| 注册表项失效 | 保留注册名并标为不可用，允许用户移除 |

状态不能只依赖颜色表达，应同时使用图标覆盖层、文字或边框状态，避免低对比度和色觉差异造成误判。

### 3.3 文本

- 页面名称：“自动进食黑名单”；
- 分类名称：“食物”“状态效果”；
- 搜索提示：“搜索名称或注册名”；
- 状态文字：“允许”“已屏蔽”；
- 空状态：“没有匹配的状态效果”。

所有文本进入本地化资源，不在界面代码中硬编码。

## 4. 过滤语义

自动进食候选必须同时满足：

```text
物品未命中食物黑名单
且
FoodProperties 中没有效果命中效果黑名单
```

建议集中为单一过滤方法，三种模式不能分别实现，以避免行为漂移。

核心读取方式：

```java
FoodProperties properties = stack.getFoodProperties(player);
```

随后遍历 `properties.getEffects()` 中的 `Pair<MobEffectInstance, Float>`，使用效果注册名匹配黑名单。不能按本地化名称存储或比较，因为语言切换会导致名称变化。

### 4.1 概率效果

只要效果在标准列表中声明，无论触发概率是否为 100%，都视为命中黑名单并跳过食物。

这是保守且可预测的黑名单语义：玩家屏蔽某效果，表示不接受获得该效果的任何可能性。

### 4.2 玩家相关属性

每次筛选时必须使用实际候选 `ItemStack` 和当前玩家调用 `getFoodProperties(player)`。不能只按 `Item` 建立全局静态缓存，因为部分模组会根据玩家状态或物品 NBT 动态返回属性。

可以缓存效果黑名单的解析结果，但不能缓存所有食物的最终效果集合，除非缓存键完整包含影响结果的玩家状态和物品数据。首版不做这种复杂缓存。

### 4.3 空属性与异常项

- `FoodProperties` 为 `null` 时不按效果黑名单拦截，后续仍由原有可食用判断决定；
- 注册表中已经不存在的效果 ID 应忽略，但保留在配置中供用户从界面移除；
- 单个模组返回异常属性时不得导致整个自动进食循环崩溃；
- 不调用 `finishUsingItem` 或相关食用方法来探测效果。

## 5. 数据与同步

保留现有物品黑名单键：

```text
rsi:food_blacklist
```

新增效果黑名单键：

```text
rsi:food_effect_blacklist
```

值使用完整效果注册名列表，例如：

```text
[
  "youkaishomecoming:drunk",
  "minecraft:nausea"
]
```

同步原则：

- 客户端负责展示和提交用户选择；
- 服务端校验注册名格式并保存；
- 服务端执行最终候选过滤；
- 玩家打开界面时同步服务端当前值；
- 未安装某个效果所属模组时，未知 ID 不导致登录或读取失败。

## 6. 兼容性调查依据

本计划依据以下本地运行依赖的字节码只读检查：

- `FarmersDelight-1.20.1-1.2.8.jar`；
- `youkaishomecoming-2.6.1.jar`。

### 6.1 Farmer's Delight

普通料理在 `FoodValues` 中通过 `FoodProperties.Builder.effect(...)` 注册效果，物品注册过程把这些属性传入 `Item.Properties.food(...)`。因此普通料理可以由通用逻辑直接识别。

已发现的特殊行为：

- `ConsumableItem` 在食用完成后调用可覆写的 `affectConsumer`；
- `MilkBottleItem` 会移除玩家现有的可治愈效果；
- `HotCocoaItem` 会从符合条件的负面效果中移除一个；
- `DogFoodItem` 还包含面向实体交互的自定义效果列表。

这些“移除已有状态”或特殊交互不属于食物要新增的标准效果。首版不把它们映射为效果黑名单，也不阻止其自定义交互。

### 6.2 Youkai's Homecoming

`EffectEntry` 保存效果 Supplier、持续时间、等级和概率，并生成 `MobEffectInstance`。普通食品和饮品最终将这些条目写入标准 `FoodProperties`，模组自己的效果提示也直接遍历 `FoodProperties#getEffects()`。

因此包括酒类声明的“酩酊”在内，大部分效果可以由通用逻辑识别。

需要注意的动态行为：

- `YHDrinkItem` 根据玩家当前的醉酒、清醒状态决定能否饮用，并在食用后调整醉酒等级；
- `FleshFoodItem` 和 `FleshSaucerItem` 根据玩家的妖怪化状态动态重建 `FoodProperties`；
- 妖怪肉会在食用后的 `consume` 逻辑中按配置、随机数和现有状态施加或延长妖怪化，并可能施加混乱；
- 部分食用行为还会触发额外事件或世界反馈。

通用过滤对动态 `getFoodProperties(player)` 中可见的效果有效，但无法完整预测 `consume` 等食用后逻辑新增的效果。

## 7. 首版兼容边界

首版承诺：

- 可靠过滤标准 `FoodProperties#getEffects()` 中声明的效果；
- 支持属性依赖当前玩家的食物；
- 概率效果按“可能触发即过滤”处理；
- Farmers Delight 普通料理和 YHK 普通食品/酒类无需专用依赖即可工作。

首版不承诺：

- 预测任意模组在 `finishUsingItem`、事件监听器或其他回调中动态添加的效果；
- 把清除已有效果视为食物效果；
- 模拟随机数判断下一次食用是否实际触发效果；
- 阻止非食用交互产生的效果；
- 绕过模组自身的饮用限制。

后续如有明确需求，可以增加可选的兼容检查器接口。专用检查器只能声明额外的潜在效果或拒绝原因，不能通过执行真实食用方法进行预测。

## 8. 实施范围

### 8.1 自动进食引擎

- 增加效果黑名单读取、写入和查询；
- 增加统一的候选食物过滤方法；
- 在 Diversity、Stack、Diet 三条路径接入同一方法；
- 保证使用实际 `ItemStack` 和当前玩家读取属性；
- 为诊断记录被物品黑名单或效果黑名单过滤的原因。

### 8.2 黑名单界面

- 增加“食物/状态效果”分类切换；
- 枚举效果注册表并显示图标和本地化名称；
- 增加按名称及注册名搜索；
- 显示允许、悬停和已屏蔽状态；
- 支持未知但已保存的效果 ID 的展示与移除；
- 保持原有食物黑名单交互兼容。

### 8.3 网络与持久化

- 扩展现有黑名单消息或新增效果黑名单消息；
- 服务端校验消息内容和玩家上下文；
- 同步界面初始状态；
- 保证旧存档没有新增 NBT 键时使用空集合默认值。

### 8.4 本地化

- 增加页面分类、搜索、状态、空状态和过滤原因文本；
- 提供中文和英文文本；
- 效果名称使用效果自身的本地化键。

## 9. 测试计划

### 9.1 单元测试

- 空效果黑名单保持现有行为；
- 无效果食物允许进入候选；
- 单一效果命中时过滤；
- 多个效果中任意一个命中时过滤；
- 概率小于 1 的效果命中时仍过滤；
- 未命中的效果不影响候选；
- `FoodProperties` 为 `null` 时不崩溃；
- 未知效果 ID 不崩溃；
- 食物黑名单和效果黑名单同时生效；
- 玩家相关 `FoodProperties` 使用传入玩家计算。

### 9.2 模式回归

分别验证以下模式不会绕过过滤：

- Diversity；
- Stack；
- Diet。

同时验证没有配置效果黑名单时，选择顺序、抽取数量、容器返还和食用事件保持不变。

### 9.3 界面测试

- 两个分类之间切换正确；
- 添加、移除并重新打开界面后状态保持；
- 中文名称搜索；
- 英文或注册名搜索；
- 已屏蔽项目排序稳定；
- 大量模组效果下滚动和点击正常；
- 不同 GUI Scale 和窗口尺寸下没有文字或控件重叠；
- 未知注册名可以安全移除。

### 9.4 实机兼容验证

- 原版带效果食物；
- Farmer's Delight 普通料理；
- Farmer's Delight 特殊饮品；
- YHK 普通食品；
- YHK 带酩酊效果的酒类；
- YHK 妖怪肉在人类、妖怪化中、妖怪化后三种状态下的属性；
- 黑名单仅影响自动进食，手动食用仍保持原模组行为。

## 10. 验收标准

功能完成需要满足：

1. 玩家可以在现有黑名单界面切换到“状态效果”并添加“酩酊”；
2. 带有该标准效果的 YHK 酒类不会被任意自动进食模式选择；
3. 移除“酩酊”后，这些酒类重新成为合法候选；
4. 配置在退出、重进世界后保持；
5. 手动食用不受限制；
6. 未安装 YHK 或 Farmer's Delight 时功能仍能正常工作；
7. 旧存档无需迁移即可加载；
8. 三种模式的回归测试、网络同步测试和界面测试通过；
9. 构建与发布 JAR 校验通过；
10. 文档中列出的动态食用后效果不被错误宣传为已完整预测。

## 11. 实施阶段与难度

### 阶段 1：过滤核心

- 建立统一效果读取与匹配方法；
- 增加数据模型和持久化；
- 接入三种自动进食模式；
- 添加核心单元测试。

难度：低。

### 阶段 2：界面与同步

- 改造现有黑名单界面；
- 增加效果列表、搜索和状态展示；
- 完成客户端与服务端同步；
- 增加界面和网络测试。

难度：中等。

### 阶段 3：实机兼容验证

- 使用 Farmer's Delight 与 YHK 验证标准及动态食物；
- 记录实际无法预测的食用后行为；
- 根据结果决定是否需要兼容检查器接口。

难度：低到中等。

总体判断：实现“屏蔽酩酊等标准食物效果”的核心难度较低；主要工作量在现有黑名单界面的清晰改造、可靠同步和三个自动进食模式的一致性测试。完整预测所有模组的动态食用后副作用不属于首版目标。
