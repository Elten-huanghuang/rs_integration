# RS 合成终端多工作站模式设计研究

## 1. 目标

在 Refined Storage（下文简称 RS）合成终端中保留原有网络物品列表，并在终端下方的工作台区域增加几个简易图标按钮，使玩家可以在以下模式间切换：

- 工作台合成；
- 锻造台合成；
- 铁砧处理。

期望效果不是跳转到另一个独立界面，而是在同一个 RS 终端内原地切换下方工作区域。

> 本文仅记录现状分析和建议实现方案，不执行代码修改。

## 2. 结论

该功能可行，但不能只替换背景纹理和槽位位置。

RS 的工作台区域由客户端界面、服务端菜单槽位、Grid 节点中的合成状态，以及拿取产物后的网络物品消耗逻辑共同组成。锻造台和铁砧使用不同的配方类型与结算规则，因此需要分别实现服务端工作站逻辑。

建议：

1. 不扩展 RS 的 `GridType` 枚举；
2. 在原有 `GridType.CRAFTING` 内增加一层 RSI 自己的工作站模式；
3. 使用固定槽位集合，通过模式控制槽位是否启用和显示；
4. 客户端按钮只发送切换请求，模式以服务端状态为准；
5. 先实现锻造台，再实现铁砧；
6. 铁砧尽量复用服务器侧原版菜单逻辑，以兼容 Forge 事件及其他模组修改。

## 3. RS 合成终端当前结构

分析目标为项目依赖中的：

- `refinedstorage-1.12.4.jar`
- Minecraft 1.20.1

### 3.1 共用 Screen 和 Menu

RS 的普通终端、合成终端和样板终端并没有分别使用完全独立的 Screen/Menu。主要共用：

```text
GridScreen
GridContainerMenu
IGrid
GridNetworkNode
```

界面和槽位根据 `IGrid.getGridType()` 分支处理。

RS 原生 `GridType` 包含：

```text
NORMAL
CRAFTING
PATTERN
FLUID
```

### 3.2 客户端背景拼接

`GridScreen.renderBackground(...)` 根据 Grid 类型选择纹理：

```text
NORMAL   -> textures/gui/grid.png
CRAFTING -> textures/gui/crafting_grid.png
PATTERN  -> textures/gui/pattern_grid.png
FLUID    -> 普通终端相关布局
```

背景不是按一个固定高度整张绘制，而是按照终端可见行数分段拼接：

1. 绘制顶部区域；
2. 按可见行数重复绘制物品列表行；
3. 绘制物品列表底边；
4. 绘制由 Grid 类型决定的底部工作区域。

因此，工作台界面实际上是 RS 终端背景底部的一部分。

### 3.3 界面高度

`GridScreen` 当前使用的关键高度为：

```text
getTopHeight() = 19

getBottomHeight():
CRAFTING = 156
PATTERN  = 169
其他     = 99
```

玩家背包的 Y 坐标由以下内容共同计算：

```text
顶部高度
+ 物品列表可见行数 × 18
+ 当前 Grid 类型对应的偏移量
```

合成终端的玩家背包额外下移 `73px`。这给 3×3 工作台和输出槽留出了空间。

### 3.4 服务端槽位拼接

`GridContainerMenu.initSlots()` 会清除并重新建立菜单槽位，然后按 Grid 类型选择附加区域：

```text
CRAFTING -> addCraftingSlots()
PATTERN  -> addPatternSlots()
```

合成终端工作台槽位的布局如下：

```text
3×3 输入槽：
x = 26 + column × 18
y = topHeight + visibleRows × 18 + 4 + row × 18

输出槽：
x = 134
y = topHeight + visibleRows × 18 + 22
```

输入槽使用：

```text
CraftingGridSlot
```

输出槽使用：

```text
ResultCraftingGridSlot
```

这说明界面纹理与真正可交互的 Slot 是两套内容，只改纹理不会改变服务端槽位和物品处理。

## 4. RS 工作台数据与配方计算

### 4.1 合成矩阵属于 Grid 节点

`GridNetworkNode` 创建并持有：

```text
TransientCraftingContainer(3, 3)
ResultContainer
CraftingRecipe currentRecipe
```

因此，RS 并不是在终端里嵌入原版 `CraftingMenu`，而是在自己的 Grid 节点中维护一个虚拟 3×3 工作台。

### 4.2 输入变化后的计算

当输入矩阵变化时，`GridNetworkNode.onCraftingMatrixChanged()` 会：

1. 检查当前缓存的 `CraftingRecipe` 是否仍匹配；
2. 不匹配时通过 `RecipeManager` 查询 `RecipeType.CRAFTING`；
3. 找不到配方时清空结果槽；
4. 找到配方时调用配方的 `assemble(...)` 生成结果；
5. 通知所有合成监听器同步菜单结果；
6. 标记 Grid 节点数据已变化。

### 4.3 拿取结果

`ResultCraftingGridSlot` 继承原版 `ResultSlot`，玩家拿取结果后会调用：

```text
IGrid.onCrafted(...)
```

随后进入 RS 的 `CraftingGridBehavior`，负责：

- 消耗工作台输入；
- 处理配方剩余物和容器物品；
- 从 RS 网络继续补充合成矩阵；
- 支持 Shift 连续合成；
- 更新网络物品追踪；
- 触发 Forge 玩家合成事件。

所以 RS 的工作台区域是一套完整的“网络化工作台”，而不是视觉组件。

### 4.4 清空矩阵

客户端清空按钮发送 `GridClearMessage`。服务端确认玩家当前打开的是 `GridContainerMenu` 后调用：

```text
IGrid.onClear(player)
```

对于合成终端，Grid 会优先把矩阵物品插回 RS 网络；网络不可用或没有权限时，则尝试返还玩家背包。

新增工作站模式时必须保持相同的物品守恒原则。

## 5. 推荐的模式模型

不要向 RS 的 `GridType` 添加新枚举值。建议在 `GridType.CRAFTING` 下增加 RSI 自己的子模式：

```java
enum TerminalWorkstationMode {
    CRAFTING,
    SMITHING,
    ANVIL
}
```

原因：

1. `GridType` 是 Java enum，无法通过普通 API 安全扩展；
2. RS 内部大量逻辑直接比较 `GridType.CRAFTING`；
3. RS 还可能生成基于 enum 的 switch 映射数组；
4. 新增 enum 值会影响纹理选择、菜单分支、能耗、Grid ID 和序列化；
5. 这些修改很容易和其他 RS 附属模组产生兼容问题。

子模式只影响合成终端底部区域，不改变终端本身仍属于 `GridType.CRAFTING` 的事实。

## 6. 客户端模式按钮

建议在工作区域上方或右侧加入三个无文字图标按钮：

```text
[工作台] [锻造台] [铁砧]
```

每个按钮应具备：

- 默认图标；
- Hover 状态；
- 当前选中状态；
- 本地化 Tooltip；
- 网络断开或终端不可用时的禁用状态。

按钮不应该直接修改客户端槽位状态。推荐流程：

1. 玩家点击按钮；
2. 客户端发送模式切换包；
3. 服务端校验当前菜单与终端类型；
4. 服务端返还或迁移旧模式输入；
5. 服务端更新模式；
6. 服务端重新计算新模式输出；
7. 将模式、输入、输出和附加属性同步客户端；
8. 客户端根据服务端模式绘制对应区域。

这样可以避免客户端和服务端对槽位含义产生分歧。

## 7. 固定槽位方案

### 7.1 推荐使用固定槽位 ID

不建议每次切换模式时删除并重新添加 Slot。

菜单打开后，客户端和服务端通过 Slot ID 处理点击、Shift 转移和同步。运行时改变槽位列表容易导致：

- 客户端点击错槽；
- 幽灵物品；
- Shift 点击目标错误；
- JEI 配方传输映射失效；
- 服务端和客户端槽位数量不一致；
- 极端情况下吞物或产生复制漏洞。

建议菜单初始化时一次性加入所有模式所需槽位：

```text
工作台：9 输入 + 1 输出
锻造台：3 输入 + 1 输出
铁砧：2 输入 + 1 输出
```

非当前模式的槽位应同时满足：

- 不绘制；
- 不响应鼠标；
- 不允许放入；
- 不允许拿取；
- 不参与 Shift 点击；
- 不接受 JEI 配方传输；
- 切换完成后不残留物品。

### 7.2 输出槽必须专用

不能让锻造台或铁砧直接复用 `ResultCraftingGridSlot`。

`ResultCraftingGridSlot` 固定围绕 `CraftingContainer`、`CraftingRecipe` 和 `IGrid.onCrafted()` 工作，锻造台与铁砧的输入消耗、经验结算和事件都不相同。

建议分别实现：

```text
TerminalSmithingResultSlot
TerminalAnvilResultSlot
```

或者使用一个根据模式委托结算的统一结果槽，但服务端结算器必须明确分离。

## 8. 锻造台模式

### 8.1 可行性

锻造台相对容易实现，适合作为第一阶段。

Minecraft 1.20.1 的锻造台输入结构固定为：

```text
锻造模板 + 基础物品 + 附加材料 -> 输出
```

它没有方块实体，也没有经验费用，因此可以自然地作为终端内部的虚拟工作站。

### 8.2 所需状态

服务端至少维护：

```text
3 个输入槽
1 个结果槽
当前 SmithingRecipe
```

### 8.3 配方计算

输入变化后应：

1. 查询 `RecipeType.SMITHING`；
2. 检查模板、基础物品和附加材料是否匹配；
3. 调用配方生成结果；
4. 将结果同步给客户端；
5. 输入不匹配时立即清空结果。

### 8.4 拿取结果

服务端应负责：

- 再次确认当前配方仍匹配；
- 每个输入槽消耗 1 个物品；
- 触发正确的配方完成、统计和事件逻辑；
- 重新计算下一次输出；
- 在支持连续制作时，从 RS 网络补充输入。

输入槽的物品校验应与配方匹配规则一致，不能只按固定物品或 Tag 粗略判断。

### 8.5 JEI 配方传输

RS 原生 `GridTransferMessage` 只允许 `CRAFTING` 和 `PATTERN` Grid，并最终进入 `IGrid.onRecipeTransfer(...)`。这套入口默认针对工作台矩阵。

锻造模式最好使用 RSI 自己的配方传输包，包中至少包含：

- 当前菜单标识；
- 目标工作站模式；
- 配方 ID；
- 三个输入位置的候选物品。

服务端必须重新查找配方并验证输入，不应直接信任客户端传来的输出。

## 9. 铁砧模式

### 9.1 可行性

铁砧也可以实现，但复杂度明显高于锻造台。

铁砧不是普通 `RecipeType` 驱动的三槽配方。它包含大量动态规则：

- 左输入与右输入组合；
- 耐久修复；
- 修复材料消耗数量；
- 附魔书应用；
- 附魔兼容性；
- 附魔等级合并；
- 物品重命名；
- `repairCost` 累积；
- 经验等级费用；
- “过于昂贵”限制；
- 创造模式例外；
- 拿取输出时扣除经验；
- 原版铁砧损坏概率。

### 9.2 建议复用服务器侧原版计算

不建议复制一份原版铁砧算法到 RSI 中。

复制算法的问题包括：

- Minecraft 或 Forge 更新后容易不一致；
- 其他模组可能通过事件、Mixin 或继承修改 `AnvilMenu`；
- Apotheosis 等模组可能改变费用上限、附魔规则或铁砧行为；
- 自己实现容易遗漏修复材料消耗量、先前惩罚等边界条件。

更稳妥的方向是：

1. 服务端创建虚拟铁砧计算上下文；
2. 尽可能让原版 `AnvilMenu` 或兼容层执行结果计算；
3. 将输入映射到虚拟菜单；
4. 读取计算后的输出、费用和材料消耗量；
5. 将只读结果同步给终端界面；
6. 玩家拿取时由服务端再次验证并结算。

### 9.3 重命名输入框

铁砧模式需要在 `GridScreen` 中增加文本框。

建议：

- 仅在铁砧模式显示；
- 使用去抖或确认式网络同步，避免每帧发包；
- 限制最大长度；
- 服务端进行合法字符和长度校验；
- 服务端重新计算费用与结果；
- 客户端只展示服务端返回的最终名称、结果和费用。

### 9.4 经验等级

经验消耗必须由服务器权威处理：

1. 计算所需等级；
2. 判断玩家是否有足够等级；
3. 判断是否达到“过于昂贵”；
4. 判断创造模式例外；
5. 拿取结果时再次确认；
6. 成功拿取后扣除等级；
7. 再消耗输入并更新结果。

绝不能先消耗输入，再发现等级不足。

### 9.5 铁砧损坏策略

终端中没有真实铁砧方块，因此原版铁砧损坏逻辑没有天然目标。

推荐第一版规则：

- 不模拟铁砧损坏；
- 保留经验消耗；
- 保留修复、附魔、重命名和过于昂贵规则；
- 在配置或 Tooltip 中明确说明“终端铁砧不会损坏”。

如果将来需要成本平衡，可以另行增加 RS 能耗或配置费用，但不要伪造不存在的方块损坏状态。

## 10. 服务端状态放置位置

可以考虑两种方案。

### 10.1 持久化到 Grid 节点

类似 RS 当前工作台矩阵，把模式和输入保存到 `GridNetworkNode`。

优点：

- 关闭再打开后输入仍存在；
- 与当前工作台行为一致；
- 有明确的终端节点归属。

缺点：

- 需要较深地修改或扩展 RS 节点；
- 无线终端、便携终端和方块终端的状态归属可能不同；
- RS 更新时兼容成本较高；
- 铁砧重命名文本、费用等状态也需序列化。

### 10.2 菜单会话状态

将锻造台和铁砧输入保存在 RSI 的菜单扩展/会话对象中，关闭菜单时全部返还网络或玩家。

优点：

- 对 RS 节点侵入更少；
- 不需要保存长期状态；
- 更容易保证模式关闭时清空；
- 更适合先做 MVP。

缺点：

- 关闭终端后输入不会保留；
- 断线、死亡和菜单异常关闭时必须可靠退款；
- 多种终端来源都需要建立一致的会话。

建议第一版采用会话状态，行为定义为：关闭界面或离开工作站模式时自动返还全部输入。等功能稳定后，再评估是否值得持久化。

## 11. 模式切换与物品守恒

模式切换是最重要的安全边界。

建议服务端切换顺序：

1. 校验玩家当前打开的是目标 RS `GridContainerMenu`；
2. 校验该 Grid 确实为 `GridType.CRAFTING`；
3. 校验玩家仍有终端访问权限；
4. 拒绝非法或重复模式值；
5. 锁定本次会话，防止同 Tick 重复切换；
6. 取出旧模式的全部输入；
7. 优先返还 RS 网络；
8. 网络放不下的剩余物返还玩家背包；
9. 背包仍放不下时在玩家附近掉落；
10. 确认旧输入已经全部有明确去向；
11. 清空旧模式输出和临时状态；
12. 更新模式；
13. 启用新模式槽位；
14. 同步完整状态。

禁止在旧输入退款未完成时直接丢弃旧状态。

同样的退款过程应覆盖：

- 点击清空；
- 切换模式；
- 正常关闭菜单；
- Grid 断电；
- 网络断开；
- 玩家断线；
- 玩家死亡；
- 菜单被服务端强制替换；
- 配置热更新导致功能被禁用。

## 12. Shift 点击与网络补充

每种模式需要独立定义 Shift 点击行为。

### 12.1 玩家背包到工作区

不能无条件塞入第一个空槽，应按模式判断：

- 工作台：维持 RS 原有规则；
- 锻造台：依次判断模板、基础物品、附加材料槽；
- 铁砧：判断左输入或右输入是否接受该物品；
- 无法明确路由时不要自动移动。

### 12.2 工作区到玩家背包

Shift 点击输入槽时应优先转移到玩家背包，或按项目既有 UX 决定插回 RS 网络。

无论选择哪种行为，都必须保证服务端统一处理且失败时原槽内容不变。

### 12.3 自动补充

锻造台可以仿照工作台，从 RS 网络补充模板、基础物品和材料，以支持连续制作。

铁砧不建议第一版支持连续处理，因为：

- 每次输出的 NBT 和 `repairCost` 都可能变化；
- 经验费用可能逐次变化；
- 重命名状态会影响结果；
- 右输入可能只消耗部分修复材料；
- 自动补充容易产生非预期批量扣经验。

## 13. 网络协议建议

建议至少包含以下数据包。

### 13.1 客户端到服务端

```text
TerminalWorkstationModeChangePacket
TerminalAnvilRenamePacket
TerminalWorkstationClearPacket（也可扩展现有清空入口）
TerminalSmithingRecipeTransferPacket
```

数据包不要直接携带可信输出或经验费用。客户端只提交意图，服务端重新计算。

### 13.2 服务端到客户端

```text
TerminalWorkstationStatePacket
```

完整状态可包含：

```text
当前模式
模式输入槽内容
结果槽内容
铁砧名称文本
铁砧经验费用
铁砧是否过于昂贵
玩家当前是否可拿取结果
状态版本号/修订号
```

状态修订号可防止延迟包覆盖新状态。

## 14. Mixin 与扩展点

按照当前 RS 1.12.4 结构，预计需要涉及以下位置。

### 14.1 `GridScreen`

用途：

- 绘制模式按钮；
- 绘制锻造台和铁砧背景；
- 绘制铁砧经验费用；
- 管理铁砧重命名输入框；
- 处理按钮和文本框输入；
- 为不同模式提供 Tooltip；
- 控制非当前模式控件的可见性。

项目目前已经存在多个 `GridScreen` Mixin，可参考现有输入拦截与附加按钮方式，但应避免不同 Mixin 对鼠标事件的优先级冲突。

### 14.2 `GridContainerMenu`

用途：

- 初始化附加槽位；
- 固定所有模式的 Slot ID；
- 控制槽位是否启用；
- 修改 Shift 点击路由；
- 同步结果槽；
- 关闭菜单时触发退款。

RS 的 `addCraftingSlots()` 是私有方法，因此可能需要：

- 在 `initSlots()` 的合适位置注入；
- 使用 Accessor/Invoker；
- 或在原有合成槽之后追加 RSI 槽位。

不要用反射调用运行时会被重映射的原版方法。

### 14.3 Grid 或会话状态

用途：

- 保存当前模式；
- 保存锻造和铁砧输入；
- 计算输出；
- 处理清空、切换和关闭；
- 与 RS 网络进行输入提取和退款。

建议尽量通过 RSI 独立组件实现，减少直接向 `GridNetworkNode` 注入大量业务状态。

### 14.4 JEI 集成

用途：

- 当前为锻造模式时接收锻造配方；
- 将三个输入槽映射到终端工作区；
- 显示缺少材料；
- 服务端实际从网络提取物品；
- 拒绝模式不匹配的传输请求。

铁砧通常不是标准 JEI 配方传输目标，可暂不纳入第一版。

## 15. 与现有项目代码的关系

项目目前已经对 RS `GridScreen` 做了多处 Mixin，包括：

- 鼠标点击与拖动处理；
- 键盘处理；
- Tooltip；
- Machine Center 侧边按钮；
- JEI 配方传输；
- `CraftingGridBehavior` 扩展。

实现新模式时需要特别注意：

1. 模式按钮点击不能被 Machine Hub 或滑动提取提前消费；
2. 铁砧文本框获得焦点时，键盘输入不能继续触发终端快捷键；
3. JEI 传输必须识别当前模式；
4. 原工作台的 `CraftingGridBehaviorMixin` 只能在工作台模式生效；
5. 关闭与清空逻辑不能只清理原有 3×3 矩阵；
6. 新按钮位置不能与 RS 原生 SideButton 和现有 Machine Center 按钮重叠。

## 16. 不推荐方案

### 16.1 只切换纹理和移动槽位

不可行。服务端仍会把输入解释为 `CraftingContainer`，并查询 `RecipeType.CRAFTING`。

### 16.2 客户端单方面切换模式

不可行。服务端槽位和客户端显示不一致，会造成错槽、幽灵物品和潜在物品异常。

### 16.3 直接扩展 `GridType`

兼容成本过高。RS 内部 enum switch、Grid ID、能耗和序列化都需要跟随修改。

### 16.4 每次切换都重建不同数量的槽位

风险较高。Slot ID 在菜单会话中不稳定，会影响点击、同步、Shift 转移和 JEI。

### 16.5 复制原版铁砧算法

不利于兼容 Forge 和其他修改铁砧规则的模组。应优先复用服务器侧原版/兼容菜单计算。

### 16.6 信任客户端提交的结果或费用

存在复制物品、免费铁砧和非法附魔风险。所有输出、费用与消耗必须由服务端重新计算。

## 17. 分阶段实施建议

### 阶段一：界面与模式协议骨架

- 增加三枚模式按钮；
- 增加服务端权威模式状态；
- 模式切换时可靠清空并返还输入；
- 保持工作台模式行为完全不变；
- 完成完整状态同步。

### 阶段二：锻造台 MVP

- 添加 3 输入和 1 输出；
- 服务端查询锻造配方；
- 正确消耗输入并生成结果；
- 支持清空、关闭和模式切换退款；
- 暂不支持 JEI 与自动补充。

### 阶段三：锻造台完整体验

- 支持 JEI 配方传输；
- 支持从 RS 网络提取输入；
- 支持连续锻造或 Shift 制作；
- 完善槽位 Tooltip 和缺料提示；
- 验证原版及模组锻造配方兼容性。

### 阶段四：铁砧 MVP

- 添加 2 输入和 1 输出；
- 添加重命名文本框；
- 复用服务器侧铁砧计算；
- 同步经验费用和过于昂贵状态；
- 拿取时服务端扣经验并结算输入；
- 不模拟铁砧损坏；
- 不支持连续处理。

### 阶段五：兼容性与打磨

- 验证 Apotheosis 等铁砧修改模组；
- 验证无线/便携合成终端；
- 验证终端断电和跨维度访问；
- 验证 JEI/REI 行为；
- 增加配置开关和本地化；
- 增加物品守恒与恶意数据包测试。

## 18. 重点测试清单

### 18.1 工作台回归

- 原有工作台配方正常；
- Shift 连续合成正常；
- 容器物返还正常；
- JEI 传输正常；
- 清空矩阵正常；
- 原有 RSI 合成扩展不受影响。

### 18.2 模式切换

- 空输入切换；
- 输入非空时切换；
- RS 网络已满时切换；
- 玩家背包已满时切换；
- 快速连续点击模式按钮；
- 延迟环境下乱序状态包；
- 切换瞬间关闭界面；
- 切换瞬间断线。

### 18.3 锻造台

- 原版升级配方；
- 原版纹饰配方；
- 模组自定义 `SmithingRecipe`；
- NBT 保留；
- 输入不完整时无输出；
- 拿取前改变输入；
- Shift 点击路由；
- JEI 候选材料选择；
- 网络材料不足时不得部分吞物。

### 18.4 铁砧

- 重命名；
- 取消重命名；
- 耐久合并；
- 修复材料消耗一个或多个；
- 附魔书应用；
- 不兼容附魔；
- 同等级附魔升级；
- 经验不足；
- 过于昂贵；
- 创造模式；
- 输出改变后的费用更新；
- 拿取结果时输入被修改；
- 其他模组对铁砧规则的修改。

### 18.5 物品守恒

对每种工作站和退出路径验证：

```text
初始网络物品
+ 初始玩家物品
+ 工作区输入
+ 已生成且合法拿取的输出
= 最终网络物品
+ 最终玩家物品
+ 合法消耗
+ 合法输出
+ 明确掉落实体
```

任何异常路径都不允许出现静默丢失或重复返还。

## 19. 最终建议

“在 RS 合成终端底部用按钮切换工作台、锻造台和铁砧”在技术上可行，且不需要创建新的 RS Grid 类型。

推荐架构是：

```text
RS GridType.CRAFTING
└── RSI TerminalWorkstationMode
    ├── CRAFTING：继续使用 RS 原逻辑
    ├── SMITHING：RSI 服务端锻造处理器
    └── ANVIL：RSI 虚拟铁砧兼容层
```

实现顺序应为：

```text
模式协议与固定槽位
-> 锻造台
-> 锻造台 JEI/自动补充
-> 铁砧
-> 跨模组兼容与守恒测试
```

其中锻造台属于中等难度；铁砧由于经验、重命名、动态消耗和跨模组规则，属于主要难点。只要坚持服务端权威、固定 Slot ID、切换先退款，以及尽量复用原版铁砧计算，该方案可以稳定落地。
